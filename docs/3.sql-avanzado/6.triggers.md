Un **trigger** (o disparador) es un objeto de la base de datos que consiste en un conjunto de acciones que se ejecutan automáticamente en respuesta a un evento específico, como una inserción (`INSERT`), actualización (`UPDATE`) o eliminación (`DELETE`) de datos en una tabla, vista o una tabla foránea, mediante cualquier evento de una sentencia DML (Data Manipulation Language), esto incluye por cierto INSERT, UPDATE, DELETE o TRUNCATE. Se utilizan para ejecutar reglas de negocio en los datos, auditar o validar datos.. Se asocian a las tablas y funcionan como "callbacks" o eventos que se activan cuando el usuario interactúa con los datos. Estos disparadores se activan cuando se modifica la data


### Características
*   **Finalidad:** Se utilizan habitualmente para **auditar cambios**, validar datos antes de que se guarden, imponer reglas de negocio complejas o mantener la integridad y consistencia entre tablas relacionadas.

*   **Momento de ejecución:** Pueden definirse para ejecutarse **antes** (`BEFORE`), **después** (`AFTER`) o **en lugar de** (`INSTEAD OF`) la operación que los origina.

*   **Alcance:** Pueden activarse una sola vez por cada instrucción (`FOR EACH STATEMENT`) o repetirse por cada fila afectada por la operación (`FOR EACH ROW`).

*   **Acceso a datos:** Los triggers permiten comparar los valores antiguos (`OLD`) de una fila con los valores nuevos (`NEW`) tras un cambio.



Estructura de un trigger

```sql showLineNumbers
CREATE TRIGGER triggername}
{BEFORE | AFTER} {INSERT | UPDATE| DELETE }
ON tablename FOR EACH ROW
codigo;
--
CREATE [OR REPLACE] TRIGGER nombre_de_trigger
BEFORE -- contexto de tiempo
INSERT -- acción
ON nombre_de_tabla
FOR EACH ROW
BEGIN
  codigo
END;
```

El contexto de tiempo aplica a una de las siguientes situaciones:

**BEFORE**: Solo se aplica a tablas y antes de verificar las restricciones. Es útil para chequear las restricciones de datos en aquellas situaciones donde no se puede verificar la integridad de ellos mediante las referencias.

**AFTER**: Solo aplica en tablas y después de realizar las operaciones. Se emplea para ejercer cambios en otras tablas en cascada.

**INSTEAD OF**: Se aplica en vistas, para hacerlas actualizables.


### Ejemplo: Sistema de Auditoría de Calificaciones
Este ejemplo muestra cómo crear un sistema que registre automáticamente cualquier cambio en las notas de los alumnos para mantener un historial de auditoría.

#### 1. Creación de las tablas necesarias
Primero creamos la tabla principal de calificaciones y una tabla secundaria donde el disparador guardará los registros de auditoría.

```sql
-- Tabla principal
CREATE TABLE grades (
    student_id bigint,
    course_id bigint,
    course varchar(30) NOT NULL,
    grade varchar(5) NOT NULL,
    PRIMARY KEY (student_id, course_id)
);

-- Tabla para el historial de auditoría
CREATE TABLE grades_history (
    student_id bigint NOT NULL,
    course_id bigint NOT NULL,
    change_time timestamp with time zone NOT NULL,
    course varchar(30) NOT NULL,
    old_grade varchar(5) NOT NULL,
    new_grade varchar(5) NOT NULL,
    PRIMARY KEY (student_id, course_id, change_time)
);
```

#### 2. Creación de la función del disparador
En sistemas como PostgreSQL, es necesario definir primero una función que contenga la lógica que se ejecutará. Esta función comparará si la nota cambió y, de ser así, insertará el registro en el historial.

```sql
CREATE OR REPLACE FUNCTION record_if_grade_changed()
RETURNS trigger AS $$
BEGIN
    -- Comprueba si el valor nuevo es distinto al antiguo
    IF NEW.grade <> OLD.grade THEN
        INSERT INTO grades_history (
            student_id, course_id, change_time, course, old_grade, new_grade)
        VALUES (
            OLD.student_id, OLD.course_id, now(), OLD.course, OLD.grade, NEW.grade);
    END IF;
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;
```

#### 3. Creación del disparador (Trigger)
Ahora asociamos la función a la tabla `grades`. Especificamos que debe activarse **después de una actualización** y por **cada fila** afectada.

```sql
CREATE TRIGGER grades_update
AFTER UPDATE
ON grades
FOR EACH ROW
EXECUTE PROCEDURE record_if_grade_changed();
```

#### 4. Prueba del disparador
Si realizamos una actualización en la nota de un alumno, el trigger insertará automáticamente una fila en `grades_history` sin que el usuario tenga que hacerlo manualmente.

```sql
-- El usuario solo ejecuta esta actualización
UPDATE grades
SET grade = 'C'
WHERE student_id = 1 AND course_id = 1;

-- El trigger habrá insertado automáticamente el cambio en grades_history
SELECT * FROM grades_history;
```

Este proceso permite que la base de datos se mantenga actualizada y auditada de forma autónoma, garantizando que todos los cambios queden registrados con su marca de tiempo correspondiente.
